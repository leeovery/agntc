AGENT: architecture
FINDINGS:
- FINDING: Duplicated ls-remote tag parsing logic between git-utils.ts and update-check.ts
  SEVERITY: medium
  FILES: src/git-utils.ts:36-42, src/update-check.ts:36-47, src/update-check.ts:162-176
  DESCRIPTION: There are three independent tag-parsing functions that parse the same `git ls-remote --tags` stdout format: `fetchRemoteTags()` in git-utils.ts, `parseAllTags()` in update-check.ts, and `parseTagCommitMap()` in update-check.ts. The first two produce identical output (tag name list) with near-identical logic (split on newline, filter `^{}`, extract ref after tab, strip `refs/tags/`). `parseTagCommitMap` does the same parse but retains the SHA. This means the `git ls-remote --tags` output format is interpreted in three places. If the parsing logic needs to change (e.g., handling a new edge case in tag refs), three functions must be updated in sync. The `fetchRemoteTags` in git-utils.ts is the canonical extraction point used by add and list-change-version-action, but update-check.ts rolls its own parsing from raw `execGit` output instead of composing on `fetchRemoteTags`.
  RECOMMENDATION: Consolidate tag parsing into git-utils.ts. Add a `fetchRemoteTagsWithCommits(url): Promise<Map<string, string>>` that returns the tag-to-SHA mapping. Derive the tag-name-only list from it. Have `checkConstrained` in update-check.ts call this shared function instead of parsing stdout directly. `checkTag` and `parseAllTags` can likewise be expressed in terms of the shared parser.

- FINDING: list-update-action does not handle constrained updates
  SEVERITY: high
  FILES: src/commands/list-update-action.ts:13-20, src/commands/list.ts:156-169
  DESCRIPTION: When the list detail view shows "Update" for a constrained plugin (`constrained-update-available`), clicking "update" calls `executeUpdateAction`, which calls `cloneAndReinstall` without passing the resolved tag or commit from the update check result. The update check already resolved the best matching tag within constraint bounds and its commit SHA (returned in the `constrained-update-available` result), but `executeUpdateAction` does not accept or forward these overrides. It will re-clone at the entry's existing ref, resulting in either a no-op or incorrect behavior. Compare with the update command (`src/commands/update.ts:162-172`) which correctly passes `{ newRef: result.tag, newCommit: result.commit }` when handling `constrained-update-available`. The list update action is missing this plumbing.
  RECOMMENDATION: Pass the `UpdateCheckResult` (or at minimum the resolved tag and commit) through to `executeUpdateAction` from the list detail loop. When the status is `constrained-update-available`, forward `newRef` and `newCommit` to `cloneAndReinstall`, matching the pattern used in `update.ts:runSingleUpdate`.

- FINDING: resolveTagConstraint in add.ts calls fetchRemoteTags twice for explicit constraints on bare add repos that have semver tags
  SEVERITY: low
  FILES: src/commands/add.ts:45-81
  DESCRIPTION: `resolveTagConstraint` has two sequential blocks: one for bare add (no ref, no constraint) and one for explicit constraint. The bare-add block resolves the latest tag, sets a derived constraint, and mutates `updatedParsed` to set a ref. The second block then checks `updatedParsed.constraint != null` -- but after the bare-add block, `constraint` on `updatedParsed` is still `null` (it's stored in `derivedConstraint`), so the second block does not fire for the bare-add case. This is functionally correct but architecturally fragile: the two blocks are coupled by the implicit contract that the first block's output shape prevents the second block from triggering. The `!= null` loose comparison adds further subtlety (it catches both null and undefined, which matters because `constraint` on some source types is typed as `null` vs possibly `undefined`).
  RECOMMENDATION: Add a comment or restructure with an early return after the bare-add path resolves successfully, making the mutual exclusion explicit rather than relying on field value propagation.

- FINDING: Downgrade guard in update.ts uses semver.clean on raw tag refs with fragile fallback
  SEVERITY: medium
  FILES: src/commands/update.ts:156-158, src/commands/update.ts:494-496
  DESCRIPTION: The downgrade guard (`gte(clean(entry.ref) ?? "0.0.0", clean(result.tag) ?? "0.0.0")`) appears in two places and uses `"0.0.0"` as a fallback when `clean()` returns null. If `entry.ref` is a non-semver string (e.g., a branch name that happened to be stored as ref before a constraint was added), `clean()` returns null, the fallback makes it `"0.0.0"`, and the comparison produces a misleading result. The constrained path should only be entered when `entry.constraint` is present, which strongly implies `entry.ref` is a semver tag -- but this is an assumption enforced by caller discipline rather than by the type system or a guard. Additionally, the same comparison logic is duplicated between `runSingleUpdate` (single-key mode) and `runAllUpdates` (batch mode).
  RECOMMENDATION: Extract the downgrade check into a named helper (e.g., `isAtOrAboveVersion(currentRef, candidateTag): boolean`) that encapsulates the clean+fallback+gte logic. Have both call sites use it. Consider returning `false` (proceed with update) rather than treating unparseable refs as `0.0.0`.
SUMMARY: The implementation's module boundaries are generally sound with clean separation between version-resolve, update-check, and command layers. The highest-impact issue is that the list's update action does not forward constrained update resolution data, which would cause constrained updates from the list UI to silently fail or re-install the current version. The duplicated ls-remote parsing and duplicated downgrade guards are structural concerns that increase maintenance risk.
