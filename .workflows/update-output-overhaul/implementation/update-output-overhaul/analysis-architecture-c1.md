- FINDING: newer-tags trailing line conflates display label with re-add command target, emitting a malformed `add` command for multi-group repos
  SEVERITY: medium
  FILES: src/update-render.ts:65-71, src/commands/update.ts:1043-1046, src/summary.ts:344-361, tests/update-render.test.ts:366-370
  DESCRIPTION: `formatNewerTagsLine(label, pinnedRef, newestTag)` takes a single `label` and uses it for TWO semantically distinct purposes — the human line prefix AND the re-add command target — building the command as `npx agntc add ${label}@${newestTag}`. Its caller (`emitCollapsedGroupSummary`) passes `label = groupLabel(group, groups)`, which for a multi-group repo is `@intent`-disambiguated (e.g. `owner/repo@v1.0`). The command then becomes `npx agntc add owner/repo@v1.0@v3.0` — a double-`@`, copy-paste-broken command. The sibling trailing surface, `renderOutOfConstraintSection` (summary.ts:344-361), handles the identical distinction correctly: it renders the prefix from `label ?? key` but builds the command from a separate BARE `repo` field. So the two group-collapse render surfaces treat the same label-vs-repo split inconsistently, and the newer-tags one is the buggy outlier. This is an interface contract that produces wrong results outside its narrow (single-group, where `label === bare repo`) context. The multi-group exact-pin newer-tags path also has no end-to-end test — the only assertion (update-render.test.ts:366-370) bakes the malformed `owner/repo@main@v2.0` output in as expected, so nothing flags it.
  RECOMMENDATION: Mirror the footer's shape: give `formatNewerTagsLine` a separate command-target parameter (the bare `repoOf(group)`, as the footer uses `info.repo`) distinct from the `@intent` display label, so the command is `npx agntc add owner/repo@<newest>` regardless of prefix disambiguation. Add a multi-group integration case asserting the bare command, and correct the unit test.

- FINDING: The group's resolved (ref, commit) is re-derived by three parallel projections across two modules instead of being owned once by GroupTarget
  SEVERITY: medium
  FILES: src/commands/update.ts:701-702, src/commands/update.ts:810-841, src/update-groups.ts:204-231, src/update-groups.ts:251-256
  DESCRIPTION: The single logical property "what ref/commit does this group land on" is projected from the `GroupTarget` union in three separate places, each re-encoding the same `constrained | branch | head | <unreachable default>` switch with the same "default is unreachable for a streamed group" caveat: `resolveEffectiveTarget` in update-groups.ts (clone `--branch` override + recorded commit), and `groupTargetCommit` + `groupTargetRef` in update.ts (header/member-line display). The commit projection is EXACTLY duplicated (`constrained → commit`, `branch/head → resolvedSha`) between `resolveEffectiveTarget.commit` and `groupTargetCommit`. The display "new ref" is computed twice and must agree for the group header move (update.ts `groupTargetRef` → `group.versionIntent` for branch/head) and each member-line move (update-groups.ts `reinstallMember` → `effectiveRef ?? entry.ref`); they coincide today only because a branch group's member `ref` equals its `versionIntent` by construction — a grouping invariant, not a type guarantee. This is mirrored layers where the domain type could own the transformation: the header's "new", the member line's "new", and the group-of-one collapsed line's "new" are the same value derived by three unsynchronised expressions straddling the update.ts / update-groups.ts seam.
  RECOMMENDATION: Have one home (GroupTarget's module, update-check.ts) expose the group's resolved display move `{ newRef, newCommit }` and, separately, the clone `--branch` override, so update.ts's header, update-groups' member-line move, and the collapsed group-of-one line all read one projection. At minimum collapse the exactly-duplicated commit projection so it cannot drift.
