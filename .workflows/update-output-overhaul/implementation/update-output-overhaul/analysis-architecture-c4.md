- FINDING: Non-actioned member wording is authored twice — a dead per-member `summary` in the outcome model plus the live group-collapse formatter — creating a drift-prone duplicate
  SEVERITY: low
  FILES: src/commands/update.ts:567-595 (splitMember), src/commands/update.ts:1017-1038 (renderOutcomeSummary), src/update-render.ts:74-103 (formatNewerTagsLine/formatCheckFailedLine/formatConstrainedNoMatchLine/formatUpToDateLine), src/summary.ts:340-341 (OutOfConstraintInfo.constraint)
  DESCRIPTION: The group-collapse redesign made all-mode render non-actioned categories (up-to-date / newer-tags / check-failed / constrained-no-match) once per group via update-render.ts formatters (emitCollapsedGroupSummary, update.ts:985-1014). But splitMember still builds a full per-member `summary` sentence for each of those four statuses — e.g. `${key}: Check failed — ${reason}`, `${key}: No tags satisfy constraint — plugin left untouched`, `${key}: Pinned to ${ref} — newer tags available (latest: ...)`, `${key}: Up to date`. Those strings are collected into `outcomes[]` but only their `status` is ever read (hasFailedOutcome + the up-to-date count filter); the sentences are never rendered and never asserted (confirmed: they occur only at their construction site). They independently re-author wording that the live formatters already own, and the two authorings have already diverged in casing/phrasing (`Check failed` vs `check failed`; `No tags satisfy constraint — plugin left untouched` vs `no tags satisfy <constraint> — left untouched`; `Up to date` vs `<N> up to date`). This is a latent maintenance hazard: a reader sees plausible display strings and may edit the dead copy, and any future change that renders outcomes directly resurrects stale wording. The same residue leaves renderOutcomeSummary (update.ts:1017) presenting as a general per-status renderer while its only call site (emitMemberLine fallback, update.ts:923) can only ever pass a bare `failed` — so every branch except `failed → p.log.error` is unreachable. A third, smaller instance of the same theme: OutOfConstraintInfo.constraint is a required field every producer and test must set but renderOutOfConstraintSection never reads (the type's own comment admits "no longer rendered").
  RECOMMENDATION: Treat update-render.ts formatters as the single home of non-actioned category wording. For the outcome model, stop authoring display sentences that never display: give the non-actioned PluginOutcome variants a lean shape (drop `summary`, or make it optional / a clearly-inert placeholder) so splitMember can't re-encode wording that lives in update-render.ts. Trim renderOutcomeSummary to the one reachable `failed` case (or inline it at emitMemberLine) so it no longer advertises unreachable per-status branches. Drop the write-only OutOfConstraintInfo.constraint field (and the values producers/tests feed it). All three are contained edits that remove drift surface without changing observable output.
SUMMARY: The group-collapse redesign correctly moved non-actioned display to per-group update-render.ts formatters, but left the superseded per-member wording behind as dead `summary` payload in splitMember, unreachable branches in renderOutcomeSummary, and a write-only OutOfConstraintInfo.constraint field — a low-severity drift/maintenance hazard, not a functional bug. Otherwise the architecture is clean: resolution/categorization split closes the check races, groupTargetFacets/formatVersionMove/resolveGuardedSourceDir/divergent-flag are each single-sourced, seams between grouping, orchestration, and rendering are well factored, and the grouped path is well covered by tests.
