- FINDING: Asymmetric routing in classifyAndCheck — branch comparison inlined, tag comparison delegated
  SEVERITY: low
  FILES: src/update-check.ts:123-134, src/update-check.ts:142-164, src/update-check.ts:170-185
  DESCRIPTION: classifyAndCheck delegates the tag path to a named helper (checkTag) but performs the branch-tip comparison inline (lines 129-134). This gives classifyAndCheck two responsibilities — remote-truth classification AND branch comparison — whereas its sibling status is a one-line dispatch. The inlined comparison (headSha === installedCommit ? up-to-date : update-available, lines 130-133) is also byte-identical in shape to checkHead's comparison (lines 154-157): both take a remote sha and an installed commit and produce the same up-to-date/update-available result. The domain operation "is the remote tip ahead of the installed commit" now lives in two places, and the classifier owns one copy inline rather than the routing decision alone. This is not a correctness issue — the sha-reuse requirement (branch must not issue a second lookup) correctly prevents a checkBranch(url) helper — but the comparison itself does not depend on any network call and could be named.
  RECOMMENDATION: Extract the tip-vs-installed comparison into a small pure helper (e.g. compareRemoteCommit(remoteSha, installedCommit): UpdateCheckResult) and have both checkHead and the branch arm of classifyAndCheck return through it. classifyAndCheck then reads as a pure three-way router (tag -> checkTag, branch -> compareRemoteCommit(headSha, ...), neither -> check-failed), restoring symmetry with the tag path and naming the intent. Optional given the small size; skip if the team prefers the inline form.
