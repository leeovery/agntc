- FINDING: Skills-only collection members are unupdatable — manifest key drops the `skills/` segment that `update` needs to relocate the source
  SEVERITY: high
  FILES: src/commands/add.ts:95-102, src/commands/add.ts:557-640, src/source-parser.ts:443-450, src/clone-reinstall.ts:352, tests/integration/workflows.test.ts:544-629
  DESCRIPTION: For a skills-only-root collection (the flag-free Vercel default added in c8), each inner skill is enumerated with the dir-relative segment `skills/<name>` but keyed in the manifest by `basename` only — `owner/repo/<name>` (memberKey, add.ts:101; integration test (d) asserts keys `owner/skills-only-repo/alpha`/`beta`, not `.../skills/alpha`). On install this is fine: the segment locates the on-disk dir while the key carries identity. But `update` resolves the re-cloned source dir purely from the manifest key via `getSourceDirFromKey(tempDir, key)`, which slices everything after `owner/repo` and joins it to the clone (source-parser.ts:443-450). For key `owner/repo/alpha` it computes `tempDir/alpha` — but the real unit lives at `tempDir/skills/alpha`. The recorded type for these members is `skill`, so derive-before-delete (replayRecordedSkill) checks `SKILL.md` at the wrong path, finds nothing, and ABORTS every update. The skill is permanently unupdatable; the only remedy is manual remove+add. Worse, the key is now indistinguishable from a genuine root-child member `owner/repo/alpha` whose dir really is `tempDir/alpha`, so `update` cannot recover the true location even in principle. Two locally-sound decisions — "identity = basename" (Identity & Naming) and "update reconstructs source from key" (existing tagless-replay machinery) — compose into a latent lifecycle bug for exactly the headline configless case. The spec's lifecycle contract ("Member entries replay by path … re-copy its own subdir", spec line 221) is silently broken because the stored key no longer encodes the subdir. (Orchestrator-verified REAL: getSourceDirFromKey at clone-reinstall.ts:352 confirms the key-based source reconstruction; a basename key cannot locate skills/<name>.)
  RECOMMENDATION: The manifest key cannot serve double duty as both basename-identity and source-relocation path when the two diverge. Either (a) persist the member's dir-relative source segment on the entry (e.g. a `sourceSubpath` field consumed by the update source resolver) so update can relocate `skills/<name>` while keeping the basename key and basename destination, or (b) key skills-only members by their true segment so the round-trip is lossless, accepting the longer key. Add an integration test that installs a skills-only collection member and then drives executeNukeAndReinstall/cloneAndReinstall through it to lock the round-trip — the current suite covers install (d) and update-abort/blocked paths but never updates a skills-only member, which is exactly why this gap survived.

- FINDING: `forcePlugin: memberHasAssetDirs` overloads an installer-intent flag to carry structural-disambiguation state on the per-member path
  SEVERITY: medium
  FILES: src/commands/add.ts:584-592, src/type-detection.ts:71-116
  DESCRIPTION: In the collection per-member loop, detection is invoked with `forcePlugin: memberHasAssetDirs`. `forcePlugin` is semantically the `--plugin` installer override ("the user asked to bundle this"); here it is repurposed as a structural signal ("this child has asset dirs, so resolve its skills-only ambiguity toward plugin"). The seam works today only because a child with ≥1 asset dir is never the skills-only-ambiguous shape forcePlugin would flip. KNOWN RECURRENCE (flagged medium in c8, discarded as "no restructuring required for correctness").
  RECOMMENDATION: Let detection own the membership-context resolution (a distinct member-context mode) rather than encoding it as a forced override at the call site.

- FINDING: `noAgentsMessage` exists as the single-source helper but the two list-action callers route the no-agents skip through `failureMessage`, which double-wraps semantics
  SEVERITY: low
  FILES: src/clone-reinstall.ts:186-211, src/commands/list-update-action.ts:51-52, src/commands/list-change-version-action.ts:102-103
  DESCRIPTION: The no-agents outcome is a lenient skip (install intact, must not exit non-zero), modelled as a non-failed status (CloneReinstallNoAgents). Both list actions treat it via isCloneReinstallFailure + failureMessage and surface it as success:false/changed:false; update.ts gives it dedicated onNoAgents handling and does not treat it as a hard failure. The list actions collapse a benign skip into the failure channel.
  RECOMMENDATION: Give the list actions a discriminated outcome distinguishing the lenient no-agents skip from a true failure (mirroring update.ts's onNoAgents), or document the intentional flattening.

SUMMARY: One high-impact composition bug introduced by the c8 skills-only enumeration: skills-only collection members are keyed by basename but `update` reconstructs the source path from that key, dropping the `skills/` segment and rendering those members permanently unupdatable (and indistinguishable from root-child members). Plus a medium flag-overloading seam (KNOWN RECURRENCE from c8) and a low no-agents-skip inconsistency across update vs list callers.
