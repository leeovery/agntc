- FINDING: Skills-only default (the headline Vercel-compatible path) resolves to an empty collection and cannot be installed
  SEVERITY: high
  FILES: src/type-detection.ts:92-96, src/type-detection.ts:151-153, src/commands/add.ts:293-305, src/commands/add.ts:486-504, src/collection-select.ts:13-15
  DESCRIPTION: The spec (Structural Type Detection → "Skills-only resolution") states the default for a root `skills/`-only repo is "collection (menu) — Vercel-compatible, the common third-party path. Works flag-free," and explicitly describes Vercel's `discoverSkills` walking one level into `skills/` and returning every `SKILL.md` as independently installable. The implementation instead returns `{ type: "collection", plugins: [] }` (type-detection.ts:96, asserted as intended in tests/type-detection.test.ts:155 and :299-305). The skills-only branch short-circuits BEFORE `scanCollectionMembers`, and `scanCollectionMembers` only ever scans the root's child dirs — so the individual skills inside `skills/` (`skills/<name>/SKILL.md`) are never enumerated as members. Downstream, `runCollectionPipeline` feeds `selectCollectionPlugins({ plugins: [] })`, which returns `[]` (collection-select.ts:13-15), and add.ts:500-503 then aborts with `p.cancel("Cancelled — no plugins selected")`. Net effect: the central anchor case of the feature — a bare `skills/`-only third-party repo installed flag-free — produces a zero-member menu and cannot be installed at all on the default path. Only the override path (`--plugin` / `type: plugin`, which bundles the whole `skills/` as one plugin) works; the spec-mandated default "menu of N skills" is missing.
  RECOMMENDATION: Make the skills-only default enumerate the skills inside `skills/` as collection members (one level into the `skills/` dir, mirroring Vercel's `discoverSkills`) so the default produces a selectable menu of `skills/<name>` units that install as bare skills. Either populate `plugins` from a `skills/` scan in the skills-only default branch, or route the skills-only-default collection through a member source rooted at `skills/`. Add an integration test that installs a `skills/`-only repo flag-free and verifies each inner skill is offered and installable.

- FINDING: `update`'s symlink-escape boundary is sound but the spec's path-traversal guard is intentionally skipped on update — confirm this is the accepted interpretation
  SEVERITY: low
  FILES: src/nuke-reinstall-pipeline.ts:92-109, src/source-parser.ts:443-450
  DESCRIPTION: The Copy-Safety section lists the path-traversal guard as in-scope and says it "runs on every copy that ingests cloned content — both `add` and `update`'s re-copy" (that sentence is specifically about the symlink guard; the path-traversal guard is scoped to "source-supplied subpath / the tree-path URL's `<subpath>`"). On update, the unit subdir is derived from the recorded manifest KEY via `getSourceDirFromKey` (source-parser.ts:443) — `join(tempDir, parts.slice(2))` — not from a fresh source-supplied selector, and that join is unvalidated for traversal. The pipeline comment (nuke-reinstall-pipeline.ts:97-102) argues the path-traversal guard is therefore unnecessary on update (the key is internally controlled, not attacker-supplied), running only the symlink scan. This is a defensible reading of "source-supplied," and the symlink-escape guard (the true content boundary) does run on update with the correct clone-root boundary. Flagging only so the divergence from a literal reading of "path-traversal … in scope for configless" is an explicit, accepted decision rather than an oversight.
  RECOMMENDATION: No code change required if the manifest is treated as trusted local state. If defence-in-depth against a hand-edited manifest key is desired, run `assertSubpathWithinClone(tempDir, parts.slice(2).join("/"))` before deriving `sourceDir` on update. Otherwise leave the documented decision as-is.
