- FINDING: `update` treats a re-cloned config of shape `{ agents: [], type: ... }` as a no-agents skip, diverging from the spec's lenient empty-agents rule and from the `add` path
  SEVERITY: medium
  FILES: src/nuke-reinstall-pipeline.ts:120-137, src/nuke-reinstall-pipeline.ts:256-269, src/config.ts:50-61
  DESCRIPTION: The spec's Agent Selection section (lines ~282-290) makes "config present but `agents: []` (empty)" one of three "no valid constraint" cases that MUST fall back to the lenient default (no restriction / all KNOWN_AGENTS), explicitly "No hard errors for config problems." `readConfig` returns a non-null `{ agents: [], type: rawType }` whenever the parsed `type` is a string but the `agents` array is empty or all-unknown. On `add` this is handled correctly: `config?.agents ?? []` feeds `selectAgents`, where `[]` triggers the KNOWN_AGENTS default. But on `update`, `resolveAgents` (nuke-reinstall-pipeline.ts:256-269) special-cases only `configAgents === undefined`; a defined-but-empty `config.agents` flows into `computeAgentChanges`, intersects the entry's recorded agents with the empty set, yields zero effective agents, and returns `{ status: "no-agents" }` — surfacing "no longer supports any of your installed agents" and skipping the update. The same config shape is lenient on add but a (benign-looking but spec-wrong) skip on update. Narrow (the re-cloned agntc.json must carry a recognised/string `type` alongside empty or all-invalid `agents`), but genuine drift from the spec's unified "empty agents → no usable constraint → lenient default" decision and an add-vs-update inconsistency. A config `{ agents: [] }` with no `type` returns null and is unaffected; only the `type`-bearing variant trips this.
  RECOMMENDATION: In `resolveAgents`, treat an empty `configAgents` array the same as `undefined` — impose no restriction, keep the entry's recorded agents. Equivalently guard `if (configAgents === undefined || configAgents.length === 0)`. Add a pipeline test covering `{ agents: [], type: "plugin" }` asserting recorded agents preserved (status success, not no-agents).
- FINDING: Update's relocation uses a stored `sourceSubpath` to join against the clone without a path-traversal containment check
  SEVERITY: low
  FILES: src/clone-reinstall.ts:362-364, src/copy-safety.ts:33-44
  DESCRIPTION: On `add`, every source-supplied subpath is gated by `assertSubpathWithinClone` before first use (add.ts:276-287). On `update`, the recorded `entry.sourceSubpath` is joined directly (`join(tempDir, entry.sourceSubpath)`) with no equivalent lexical containment assertion before the pipeline reads at that path. The comment at nuke-reinstall-pipeline.ts:99-102 notes update relies solely on the symlink scan because it "replays a recorded manifest key ... not a fresh source-supplied selector" — reasoning that held before cycle-9 added `sourceSubpath`, now a second source-derived path component fed into the join. In practice the value is agntc-internally derived (`skills/<name>`, never `..`) and the symlink guard still runs, so exploitability is effectively nil — hence low. But the add path's defense-in-depth for the one varying segment is no longer mirrored on update.
  RECOMMENDATION: Optional hardening only — call `assertSubpathWithinClone(tempDir, entry.sourceSubpath)` in the remote branch of `cloneAndReinstall` before the `join`, mirroring add step 2c. No behavioural change for valid manifests.

One medium spec drift: `update`'s `resolveAgents` treats a defined-but-empty `agents` array as a no-agents skip rather than the spec-mandated lenient default, making add and update inconsistent for `{ agents: [], type }` configs.
