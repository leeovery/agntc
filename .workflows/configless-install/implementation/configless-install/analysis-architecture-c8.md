- FINDING: Collection-member skills-only resolution reuses `forcePlugin` as a membership signal, coupling two orthogonal axes
  SEVERITY: medium
  FILES: src/commands/add.ts:560-568, src/type-detection.ts:79-108
  DESCRIPTION: In the collection pipeline, a member's type is resolved with `detectType(pluginDir, { configType, forcePlugin: memberHasAssetDirs })`. The `forcePlugin` input is the spec's `--plugin` installer override — its sole sanctioned job (per the spec's "Selector / `--plugin` orthogonality" and "`--plugin` scope") is to resolve the skills-only ambiguity for the user-selected unit. Here it is repurposed as a structural membership disambiguator ("a skills-only child with an asset dir is a plugin member, not a nested collection"). The two concerns are made to ride the same parameter, and correctness depends on a non-obvious invariant: `memberHasAssetDirs` must be true only for the skills-only-with-skills-dir case and false for a member-dirs child (which would hard-error under `forcePlugin`). The inline comment spends ~12 lines defending why this happens to be safe — a signal the seam is carrying more meaning than its name admits. A member-dirs child has zero asset dirs at its own root so it stays false today, but the safety is an emergent property of `findPresentAssetDirs` semantics rather than an expressed contract; a future change to what counts as an "asset dir" or to member structure could silently turn a membership decision into a spurious `TypeConflictError`. The override layer (`detectType`) cannot tell a deliberate user `--plugin` from this internal membership nudge.
  RECOMMENDATION: Express member-level skills-only resolution as its own structural decision rather than threading it through the installer-override parameter. Either add a distinct, named option to `DetectTypeOptions` (e.g. `resolveSkillsOnlyAs: "plugin" | "collection"`) that the membership scan sets explicitly, or have `detectType` accept the already-computed structural-kind intent for the one-level-down case. This keeps `forcePlugin` meaning only "the user passed --plugin" and makes the membership rule self-documenting instead of relying on a coincidental asset-dir count.

- FINDING: Update replay re-implements the copy-dispatch seam that `copyUnit` already owns
  SEVERITY: low
  FILES: src/nuke-reinstall-pipeline.ts:174-245, src/copy-unit.ts:34-73
  DESCRIPTION: `copy-unit.ts` was introduced as the single seam that maps a resolved standalone unit to its copy call (`copyBareSkill` vs `copyPluginAssets`) and assembles `copiedFiles`/`assetCountsByAgent`, explicitly "shared by the standalone and collection-member install paths." The update pipeline's `replayRecordedSkill`/`replayRecordedPlugin` does not use it — it calls `copyBareSkill`/`copyPluginAssets` directly and rebuilds the same two-arm dispatch. The recorded `type` ("skill" | "plugin") maps onto the `StandaloneDetected` discriminant the seam already understands, so the replay arms and the add arms are logical mirrors that can drift (e.g. if the bare-skill copy ever needs to surface counts, or the plugin copy gains a new return field, two sites must change). The divergence is partly justified — recorded-plugin replay needs the live `findPresentAssetDirs` scan to pick which dirs exist now, which the add path gets from `detected.assetDirs` — but the copy step itself (after the asset dirs are known) is identical to `copyUnit`'s plugin arm.
  RECOMMENDATION: After the derive-before-delete predicate passes, route the actual copy through `copyUnit` by constructing the `StandaloneDetected` value the replay has already validated (`{ type: "plugin", assetDirs: presentAssetDirs }` or `{ type: "bare-skill" }`). The validation/abort gate stays in the pipeline; the copy dispatch collapses to the one seam, so a future change to copy semantics is a single edit.

- FINDING: Two independent type-derivation authorities (`manifestTypeFromDetected` vs `deriveTypeFromFiles`) can disagree on the same unit
  SEVERITY: low
  FILES: src/manifest.ts:59-63, src/manifest.ts:111-129
  DESCRIPTION: The persisted `ManifestEntry.type` is produced by two unrelated functions: `manifestTypeFromDetected` (structure → type, for fresh installs) and `deriveTypeFromFiles` (installed files → type, for legacy backfill). The spec deliberately makes these non-composable (backfill must read local files, never re-derive from structure — the anti-drift rationale), so this is not a missed-composition defect. The architectural note is that the two encode the same `skill | plugin` distinction via different evidence and could classify the same on-disk unit differently: a single-skill bundle installed fresh today records `plugin` via `manifestTypeFromDetected`, but the identical install backfilled from a legacy manifest records `skill` (the spec's accepted single-skill collateral). The spec accepts this, and replay behaviour is identical, so no bug — but the two derivation paths are an inherent coherence seam worth keeping visible.
  RECOMMENDATION: No restructuring required given the spec's anti-drift mandate. To keep the seam from silently drifting, add a shared doc-anchor (or a small co-located comment cross-referencing both functions) stating that the two are the only two type-authorities and must stay semantically aligned on the skill/plugin boundary. Optionally, a single round-trip test asserting that a fresh multi-asset install's recorded type equals the backfill of its own files would catch accidental divergence in CI.

SUMMARY: The implementation composes cleanly overall — discriminated-union results, a well-isolated copy-safety module, and a single `mapCloneFailure` dispatch seam shared across the four reinstall entry points show strong seam discipline. The notable structural concern is the collection-member path overloading the `--plugin` installer-override parameter to carry a structural membership decision (medium); the update pipeline re-implementing the `copyUnit` dispatch seam and the two parallel type-derivation authorities are lower-impact coherence seams worth keeping visible.
